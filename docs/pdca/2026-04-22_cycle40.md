# PDCA cycle40 — Detector threshold sweep + regime emission analysis

**Date:** 2026-04-22
**Parent:** `docs/pdca/2026-04-22_cycle39.md` (4 router variants all byte-identical to default child).
**Goal:** with PR #138 (`detector_config` exposed), sweep `TrendADXMin × VolatileATRPercentMin` via WFO and answer "is regime routing useful at all on LTC 15m?".

## Enabler — PR #138

Adds `detector_config` block to `RegimeRoutingConfig` schema + thread to `regime.NewDetector` + 3 WFO override paths (`regime_routing.detector_config.{trend_adx_min, volatile_atr_percent_min, hysteresis_bars}`). 15 unit tests including a wiring-confirmation test that the same indicator bar driven through two routers with different `TrendADXMin` produces different committed regimes.

## Cycle 40b — diagnostic: which regime is the detector actually committing?

To rule out a wiring bug vs. a data-driven invariant, ran four "diagnostic" routers, each with `default = sl6` and a single override on a different regime label. If a given regime fires often, the override (sl14) wins; if it never fires, the default (sl6) wins.

| diagnostic profile | override regime | 1yr_2025 result | matches |
|---|---|---|---|
| router_diag_bull | bull-trend | +6.38% / 3755 trades | sl6 (default) |
| router_diag_bear | bear-trend | +6.38% / 3755 trades | sl6 (default) |
| router_diag_range | range | +6.38% / 3755 trades | sl6 (default) ⚠️ |
| router_diag_volatile | volatile | +6.38% / 3755 trades | sl6 (default) |

All four produce sl6's numbers, **including diag_range** even though the detector emits `range` 99.9% of the time (see histogram below).

This is **not** a detector wiring bug — it surfaces the **same PR-5 part E limitation already documented in #136**: the runner's `tickRiskHandler` is constructed once with one fixed `RiskConfig` (the default child's SL=6/TP=6 + trailing). Even when the router successfully delegates signal generation to sl14 during range bars, **exit logic uses the default child's SL/TP**, so the resulting trade pattern still looks like sl6.

## Cycle 40c — regime histogram on the real LTC stream

Driven by `BACKTEST_REAL_DATA=1 go test … TestDetector_RealLTCStream_RegimeHistogram` against the committed 120,269-row LTC 15m CSV. ADX/ATR computed exactly the way the backtest runner does, three configs compared:

| config | range | volatile | bull-trend | bear-trend | unknown |
|---|---:|---:|---:|---:|---:|
| **defaults** (ADX≥20, ATR%≥2.5, hysteresis=3) | **99.9%** (120,048) | 0.1% (161) | 0% | 0% | 0% |
| loose (ADX≥10, ATR%≥0.5) | 89.7% (107,829) | 10.3% (12,380) | 0% | 0% | 0% |
| tight (ADX≥50, ATR%≥5.0) | 100.0% (120,193) | 0.0% (16) | 0% | 0% | 0% |

**No threshold combination produces a single bull-trend or bear-trend bar over 120k bars (~3.4 years of LTC 15m candles).** ADX on LTC 15m simply does not cross the floor required to commit a directional regime, even when the floor is 10 (Wilder's "weak trend" cutoff).

Drilling into why: the cycle28-37 finalists derive their alpha from the *underlying* `ConfigurableStrategy`'s SMA / EMA / MACD / RSI signal stack, which fires aggressively on small directional moves that ADX never registers as "trend". On a 15m timeframe with frequent reversals, ADX stays compressed in the 5-15 band almost continuously.

## Verdict — hard reject regime routing on LTC 15m

The conjunction of three findings closes the question:

1. **WFO sweep**: 20 distinct (TrendADXMin × VolatileATRPercentMin) cells produce **identical** results per window. The detector axis is dead.
2. **Diagnostic routers**: even when an override sits on the regime that does fire (range), the result still matches the default — because the **part E risk-envelope limitation** prevents the override from changing trade exits.
3. **Regime histogram**: 0% directional regime emission across 120k LTC 15m bars under any threshold combination. Volatile peaks at 10% only when the volatility floor is dropped to 0.5%.

Even if PR-5 part E (per-regime tickRiskHandler) were implemented today, **bull-trend and bear-trend never fire**, so the only routing axis that could ever be active is range vs. volatile. That is a single binary split, not the four-regime decomposition the design assumed. Useful product features tend not to come from a single binary axis on an indicator that takes 3 years of data to disagree with itself.

**Decision:** stop the regime-routing work. The infrastructure (PRs #133/134/135/136/138) is correct and stays in main as a foundation that other assets / timeframes might exercise (BTC 1h / ETH 4h might emit trend regimes), but on **LTC 15m specifically, regime routing is not the path to v5 promotion**.

## What this leaves on the table

- **Best aggressive candidate**: `experiment_2026-04-22_sl14_tf60_35` from cycle28-37. gM +16.22% on 2023-04..2026-03, worst +9.26%, 3yr DD 6.01% — but **−57.97% on 2022 bear**.
- **Best defensive candidate**: `experiment_2026-04-22_sl6_tr30_tp6_tf60_35` from cycle28-37. Survives both 2022 (+28.30%) and 2023-26 (+14.69%), worst −7.32% (2024 chop), 3yr DD 24.5% (just over the 20% target).
- **No router beats either of these on any period.**

The user-facing question becomes: "ship the aggressive (assume no 2022-style bear) or the defensive (accept smaller upside) finalist as v5, with a written caveat — or keep iterating on a non-routing axis?"
