## Root cause ŌĆö the detector never emits bear-trend / volatile

Three independent pieces of evidence converge:

1. v5a (override on bear+volatile) ŌēĪ v5b (override on bear only) ŌēĪ v5c (override on range only). For all three to be identical to sl14, the detector must commit `bull-trend` continuously ŌĆö none of bear-trend / volatile / range ever fires.
2. Same outcome on the no-HTF run, so the issue is not just HTF SMA cross ŌĆö primary SMA20/SMA50 also stays cross-positive in the windows tested.
3. The no-HTF `old3yr` test reproduces both numbers (sl14 ŌłÆ57.97%, sl6 +28.30%) ŌĆö the only way for v5a-c (default sl14) and v5d (default sl6) to land on those exact numbers is "router always picks default".

In retrospect this was predictable: the detector's direction logic prioritises HTF Ichimoku ŌåÆ HTF SMA20/SMA50 ŌåÆ primary SMA20/SMA50, and on LTC 15m the SMA20-vs-SMA50 cross flips slowly enough that long stretches commit bull-trend. The current ADX threshold (default 20) and ATR% threshold (default 2.5%) further bias the classifier toward `bull-trend`/`bear-trend` over `range`/`volatile` whenever either SMA is on top.

## Cycle 39b ŌĆö would tuning the detector save it?

Not without a code change. `regime.Config{TrendADXMin, VolatileATRPercentMin, HysteresisBars}` is a Go struct ŌĆö it is **not** plumbed through `RegimeRoutingConfig` and not surfaced in the JSON profile. Without per-profile detector config:

- A user-tunable threshold would need a new schema field (`detector_config` block in `regime_routing`), a builder change to thread it into `regime.NewDetector`, and updated WFO override paths so a sweep can find good thresholds.
- Even with that field, a single set of thresholds is unlikely to be right across 2022 (deep bear) and 2024 (chop) and 2025 (mild bull). Per-asset / per-regime calibration is the real fix.

This is a meaningful PR-5 follow-up but does not belong in cycle39's scope. Tracked as **PR-5 part F (detector config in profile schema + WFO override path)**.

## Conclusion

**The cycle39 router profiles do not clear the v5 promotion bar.** They are functionally indistinguishable from one of the two finalists they wrap ŌĆö the routing infrastructure works (PR #135's three wiring tests verify this on synthetic data and stub strategies) but the **detector defaults are too restrictive for LTC 15m to ever swap profiles in practice**.

Restating the v5 promotion requirements from `docs/pdca/README.md`:
- 1yr/2yr/3yr all > 0 ŌĆö **failed** for v5a/b/c (sl14 lineage's ŌłÆ58% on 2022); **failed** for v5d (sl6's ŌłÆ7.32% on 2024).
- DD Ōēż 20% ŌĆö failed for v5a/b/c on old3yr (91%); failed for v5d on old3yr (24.5%).
- Improvement over both finalists ŌĆö **none of the four routers improves on either finalist on any period**.

## What we proved end-to-end

1. **PRs #133-#135-#136 are correctly wired.** v5d using sl6 as default and v5a-c using sl14 as default produce sl6's and sl14's numbers respectively to the trade. The handler-runner-router-detector chain delivers signal generation per profile.
2. **Detector defaults are too coarse to differentiate LTC 15m regimes.** With `TrendADXMin=20` and `VolatileATRPercentMin=2.5`, the bull-trend branch wins essentially 100% of the time on this asset/timeframe.
3. **Per-regime SL/TP differentiation is structurally not possible** until PR-5 part E (tickRiskHandler refactor). Even when overrides eventually fire, today's tickRiskHandler will use one fixed RiskConfig (the default child's) for the whole run.
